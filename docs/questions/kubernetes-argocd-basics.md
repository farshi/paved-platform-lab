# Kubernetes And Argo CD Basics

Use this as a first-pass question flow before the deeper operator questions. It starts with basic Kubernetes objects, then moves into CRDs and Argo CD.

## Mental Model

```text
kubectl
  -> Kubernetes API server
      -> built-in resources
          Namespace, Pod, Service, Deployment, ConfigMap
      -> CRD resources
          Argo CD Application
          Kyverno ClusterPolicy
          Prometheus ServiceMonitor

Argo CD Application
  -> reads a Git repo path
  -> compares Git desired state with live cluster state
  -> syncs Kubernetes resources
```

## 1. What is `tenant-a`?

`tenant-a` is a namespace, not a pod.

Check:

```sh
kubectl get namespace tenant-a
kubectl get pods -n tenant-a
kubectl get all -n tenant-a
```

In this lab:

```text
tenant-a       = namespace
demo-api       = Deployment and Service name
demo-api-xxxxx = Pod name generated by the Deployment
```

## 2. Why does `get service` show 1 but `get pods` shows 2?

A Service is one stable network entrypoint. Pods are the running copies behind it.

```text
Service demo-api
  -> Pod demo-api-a
  -> Pod demo-api-b
```

The Deployment asks for two replicas:

```sh
kubectl get deployment demo-api -n tenant-a
```

The Service routes to those pods by labels:

```sh
kubectl describe service demo-api -n tenant-a
kubectl get pods -n tenant-a --show-labels
```

## 3. How do I get nodes?

Nodes are the machines or containers where pods run.

```sh
kubectl get nodes
kubectl get nodes -o wide
kubectl describe node <node-name>
```

In k3d, nodes are Docker containers running k3s.

## 4. Does `kubectl` talk to nodes directly?

Usually no. `kubectl` talks to the Kubernetes API server.

```text
kubectl
  -> API server
  -> controllers/scheduler/kubelet
  -> nodes and pods
```

Check your current API endpoint:

```sh
kubectl config current-context
kubectl config view --minify
```

## 5. What is a CRD?

CRD means Custom Resource Definition.

Kubernetes has built-in resource types:

```text
Pod
Service
Deployment
ConfigMap
Namespace
```

A CRD lets a tool add new resource types.

Examples in this lab:

```text
applications.argoproj.io       -> Argo CD Application
clusterpolicies.kyverno.io     -> Kyverno ClusterPolicy
servicemonitors.monitoring.coreos.com -> Prometheus ServiceMonitor
```

Useful commands:

```sh
kubectl get crd
kubectl api-resources
kubectl explain application
```

## 6. Why does Kyverno install many CRDs?

Kyverno is one product, but it supports many custom resource types.

That is why you may see many Kyverno CRDs:

```text
clusterpolicies.kyverno.io
policies.kyverno.io
policyexceptions.kyverno.io
updaterequests.kyverno.io
```

In this lab, the main Kyverno resource you use is:

```sh
kubectl get clusterpolicies
```

Those policies enforce guardrails such as approved image registries, resource requests/limits, and non-root containers.

## 7. What does `kubectl get applications` show?

It shows Argo CD `Application` custom resources.

`applications` comes from this Argo CD CRD:

```text
applications.argoproj.io
```

Commands:

```sh
kubectl get applications -n argocd
kubectl get applications -A
kubectl get applications -n argocd -o wide
```

This does not show Kyverno. Kyverno has different resource types:

```sh
kubectl get clusterpolicies
kubectl get policyreports -A
kubectl get pods -n kyverno
```

## 8. How are the 4 Argo CD apps introduced?

This command creates the Argo CD `Application` resources:

```sh
make argocd-apps
```

Flow:

```text
argocd/apps.json
  -> scripts/argocd/render-apps.js
  -> kubectl apply -f -
  -> Application resources in namespace argocd
```

Expected apps:

```text
platform-guardrails-tenant-a       -> examples/tenant-a
platform-guardrails-tenant-b       -> examples/tenant-b
platform-guardrails-policies       -> policies/kyverno
platform-guardrails-observability  -> observability
```

Show them:

```sh
kubectl get applications -n argocd
node scripts/argocd/render-apps.js
```

## 9. What happens when I press Sync in Argo CD?

Sync means make the live cluster match Git.

Argo CD reads:

```text
https://github.com/farshi/paved-platform-lab.git
```

Then applies the configured path for that app.

Example drift demo:

```sh
kubectl scale deployment demo-api -n tenant-a --replicas=1
kubectl get deployment demo-api -n tenant-a
```

Argo CD should show the app as out of sync. Press Sync, then check:

```sh
kubectl get deployment demo-api -n tenant-a
```

The live state should return to the desired state from Git.

## 10. What is the Argo CD tracking annotation?

Example:

```yaml
argocd.argoproj.io/tracking-id: >-
  platform-guardrails-observability:/ConfigMap:observability/demo-api-dashboard
```

Meaning:

```text
Argo CD Application: platform-guardrails-observability
Resource kind: ConfigMap
Namespace: observability
Resource name: demo-api-dashboard
```

Argo CD uses this metadata to know which app owns which live object.

If you remove the annotation manually, the object is not deleted. Argo CD may add it back on the next sync or show drift. To remove a resource from Argo CD management, remove the manifest from Git or change the Argo CD app path.

## 11. How do I log in to Argo CD?

Open the UI:

```sh
make tools-up
```

Then open:

```text
https://localhost:18080
```

Login:

```text
username: admin
password: output from make argocd-password
```

Command:

```sh
make argocd-password
```

```sh
make build
make install-addons
make deploy
make argocd
make tools-up
```

Then open the portal:
